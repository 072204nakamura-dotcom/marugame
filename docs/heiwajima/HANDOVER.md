## 4. 採点仕様（これが全て。追加・削除禁止）

### 4-1. 判定順序
```
①レース名で企画判定（judge_kikaku.py）→ ②通常戦のみレース番号点 → ③選手表JOINで加点 → ④合計点とフォーム出力
```

### 4-2. レース単位の点（judge_kikaku.py が返す）

| 分類（レース名部分一致・優先順） | 点 | フラグ／フォーム |
|---|---|---|
| `ベイラン` → 東京ベイランチ | 0 | **鉄板・穴買い禁止／見送り推奨** |
| `ベイブレ` → ベイブレイク | +1.0 | カド攻め／4まくり(4-1・4-流し) |

**判定順序は仕様の一部**（「準優」を「優勝」より先）。judge_kikaku.py を変更したら `python3 judge_kikaku.py` でテスト11/11を必ず通す。

### 4-3. 選手表JOINによる加点（regnoで結合・進入は枠なり仮定）

| 条件 | 点 | 使う表・列 |
|---|---|---|
| 1号艇の採用スローST > 0.18（スロー遅れ●） | +1.0 | hw_st.csv `採用スローST`/`スロー遅れ` |
| 2号艇が壁弱●（hw_wall.csv `壁`列） | +1.0 | hw_wall.csv |

| 1号艇が飛ぶ型（hw_zanshi.csv `型`） | +0.5 | hw_zanshi.csv |

### 4-4. 買い目方針テキスト（2号艇の壁タイプで分岐）

| 2号艇のタイプ | 出力する方針 |
|---|---|
| 食う型 or 差し巧者● | 本線 **2-1-全(4点)**＋押さえ 2-3-1・2-4-1 |
| 素通し型 | 2アタマは買わず **3-1-全／4-1-全**、万舟狙いは X-X-1（1号艇3着沈み） |
| 壁強● | **見送り推奨**（イン65.5%・万舟10.9%） |
| 印なし | 標準：2-1-全＋3-1-全（10点） |

カバー実測（差し系決着546R条件付き）：2-1-全=19.8%/回収233%、+2-3-1・2-4-1=24.4%/208%、+3-1-全=34.2%/172%。**「条件付き」の注意書きをページに必ず載せる**（差し系になるのは崩れゾーンでも3割前後）。

### 4-5. 参考表示（採点しない・注記つきで表示のみ）

- 1号艇モーターの二連率（hw_motor_current.csv。「n不足・機力は平和島では弱い」注記）
- 当日風予報が北系3m+のとき：「**北風3m+：1号艇のST優位が痩せる・まくり差し警戒（3-1/4-1/5-1系）**」（風データが取れる場合のみ。取れなければ省略可）

## 5. daily_build_hw.py の仕様（新規作成）

毎朝1回実行。手順：

1. **当日のBファイル取得**：`https://www1.mbrace.or.jp/od2/B/{YYYYMM}/b{YYMMDD}.lzh`（UA=Mozilla/5.0、lhafile解凍、Shift-JISデコード）。`04BBGN`〜`04BEND` ブロックが無ければ「本日非開催」ページを出して正常終了。
2. **出走表パース**（平和島ブロック内）：
   - レースヘッダ：NFKC正規化後 `^\s*(\d{1,2})R\s+(\S+)` でレース番号とレース名。
   - 選手行（固定幅）：`ln[0]`=艇番(1-6)、`ln[2:6]`=登番、`ln[6:10]`=氏名、`ln[16:18]`=級（A1/A2/B1/B2）。
   - モーター番号：4つの率の後 → 正規表現 `\d+\.\d\d\s+\d+\.\d\d\s+\d+\.\d\d\s+\d+\.\d\d\s+(\d+)\s+(\d+\.\d\d)`（group1=モーターNo）。
3. **判定**：judge_kikaku.py → §4-3 の表JOIN → 合計点・フォーム・方針テキスト。
4. **HTML生成**：既存会場（tsu）の体裁を踏襲。1レース1カード。表示要素＝R番号・レース名・分類バッジ・合計点（★表示等）・選手ごとの印（壁弱●/差し巧者●/スロー遅れ●/飛ぶ型）・買い目方針・参考表示。ページ上部に会場サマリ（「差し水面／イン46%／主戦場7R・3R／企画はレース名判定」）。
5. **アーカイブ追記（任意・推奨）**：前日のKファイルを取得し `hw_races_archive.csv`／`hw_entries_archive.csv` に追記（scripts/parse_hw.py のロジック流用）。**6表の再計算は毎日不要**——月1回で十分（§7）。まずはレースカード生成だけで公開してよい。

失敗時：例外で落とさず、前回のindex.htmlを残したまま「更新失敗（日時）」を小さく表示するのが既存方針。

## 6. GitHub Actions（heiwajima.yml）

- **先に既存ワークフローの cron 一覧を確認**し、既存8場と重ならない分を選ぶ（JST 07:00台＝UTC 22:00台前日）。例：既存が 22:00/22:05/22:10… なら空きの `22:1x` を使う。
- 手動実行用に `workflow_dispatch:` も付ける。
- 手順：checkout → Python 3.x → `pip install lhafile` → `python3 scripts/heiwajima/daily_build_hw.py` → 変更があれば commit & push（既存会場と同じ書式）。
- mbrace は遅い（1ファイル約10秒）。タイムアウトは余裕を持たせる（10分以上）。
